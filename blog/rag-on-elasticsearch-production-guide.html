Building RAG on Elasticsearch: a production engineering guide

Most RAG prototypes work fine. Most RAG systems in production do not. Here is what changes between demo and deployment and how to build retrieval pipelines that hold up at scale.

Why prototypes break in production

Building a RAG prototype on Elasticsearch is straightforward: ingest a few hundred documents, embed them, run kNN queries, pass the results to an LLM. It works in an afternoon. The problems start at scale, under real query patterns, with data that changes, users who ask unexpected questions, and latency requirements that do not allow for slow retrieval.

The gap between prototype and production is almost never about the LLM. It is about the retrieval layer. Specifically: chunk quality, index design, query architecture, and the fallback logic when semantic search does not find what it needs.

Chunking strategy matters more than the model

Most RAG failures trace back to poor chunking. The default approach — split every document into fixed-size chunks of 512 tokens — works well enough for demos but breaks down for real content. Technical documentation, support tickets, and internal knowledge bases have structure that fixed-size chunking destroys.

  • Use semantic chunking. Split at logical boundaries — paragraphs, headings, list items — rather than arbitrary token counts.
  • Store chunk context. Each chunk should carry metadata about its parent document, position, and section. This allows the LLM to reason about where the information comes from.
  • Overlap aggressively for dense content. A 20% overlap on technical content significantly reduces the chance of splitting a critical sentence across two chunks.

Hybrid search is not optional

Pure vector search fails on exact-match queries. A user asking for a specific error code, a product SKU, or a proper noun will get poor results from semantic search alone because the model encodes meaning, not literals. Hybrid search — combining BM25 keyword matching with kNN vector retrieval and using Reciprocal Rank Fusion to merge the results — handles both cases reliably.

Elasticsearch supports hybrid search natively as of version 8.9 through the rank parameter in the search API. This is the single most impactful change you can make to a struggling RAG retrieval pipeline.

Latency budgets and retrieval pipeline design

A production RAG pipeline has a latency budget. The LLM call typically takes 1–4 seconds. That means retrieval needs to complete in under 200ms at p95 to keep end-to-end response times under 5 seconds. Achieving this requires: properly sized kNN parameters (num_candidates should not be set higher than necessary), warm indices on fast storage, and query-time inference only when the document count justifies it.

If you are running ELSER inference at query time on a large index and wondering why retrieval is slow, the answer is almost always that inference is happening on too many candidate documents before the top-k filter is applied. Move inference to ingest time and pre-compute sparse vectors instead.

Testing retrieval quality

The most neglected part of production RAG is measurement. Most teams judge quality by asking a few questions manually. Production-grade RAG requires a test set — a collection of representative queries with known good answers — and a retrieval evaluation metric like NDCG or MRR that you can run automatically against your pipeline as the index and queries evolve.
