ClickHouse query performance: the five mistakes that cost you seconds

Most slow ClickHouse queries share the same root causes. Here are the five patterns we see most often in production — and how to fix them without a schema rewrite.

1. Wrong sort key for your query patterns

The sort key (ORDER BY in MergeTree) is the single most impactful decision in a ClickHouse schema. It determines which columns ClickHouse can use to skip granules during a query. If your sort key does not match your most common filter predicates, ClickHouse has to scan far more data than necessary.

The fix is not always a schema rewrite. In many cases, you can create a projection that defines a secondary sort order for a specific query shape, giving you fast reads on both paths without duplicating the full table.

2. Selecting more columns than you need

ClickHouse is a columnar store. Reading a column you do not need has a real cost — it pulls additional granules off disk and into memory. The most common culprit we see is SELECT * in analytical queries that only need 3–4 fields out of a 50-column table.

  • Always name the columns you need explicitly
  • Use EXPLAIN to verify column reads in your query plan
  • Consider materialized views that pre-aggregate common column subsets

3. High-cardinality GROUP BY without pre-aggregation

Aggregating over a column with millions of distinct values at query time is expensive. If you are running the same aggregation repeatedly — daily active users, event counts by tenant, revenue by SKU — you should be pre-aggregating with a SummingMergeTree or AggregatingMergeTree and a materialized view that populates it on ingest.

This pattern shifts the cost from query time to ingest time, which is almost always the right trade-off for analytical workloads.

4. Not using skip indexes for low-selectivity filters

When you filter on a column that is not in your sort key, ClickHouse has no way to skip granules unless you have a skip index defined. A simple minmax skip index on a timestamp or numeric range column can eliminate large portions of a scan for almost zero storage overhead.

5. INSERT storms fragmenting parts

ClickHouse merges data parts in the background. If you are inserting too frequently in small batches, you end up with hundreds of small parts that have not merged yet. Queries against fragmented tables are slow because ClickHouse has to read from many parts instead of a few large ones.

The fix is to batch inserts: aim for at least 100–500ms between insert operations, or use the async_insert setting to let ClickHouse handle the batching on the server side. This alone resolves the majority of performance complaints we see on write-heavy deployments.
